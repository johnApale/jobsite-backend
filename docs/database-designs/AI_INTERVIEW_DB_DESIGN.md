# AI Interview Service — Database Design

Standalone microservice database. This is **not** part of the modular monolith's tenant DB — it's a separate PostgreSQL database owned by the AI Interview Service. Unlike the monolith (which uses database-per-tenant), this service uses a **single shared database with tenant ID filtering** because provisioning a separate AI interview database per tenant would be overkill for a stateless, compute-heavy service.

The AI Interview Service generates interview questions tailored to job requirements, presents them to candidates, scores responses using an AI provider (OpenAI, Anthropic, or Azure OpenAI), and produces a comprehensive evaluation. It communicates with the monolith exclusively via message broker — no direct database access.

**Inbound:** `CandidateReadyForInterviewEvent` from Screening (via message broker)
**Outbound:** `InterviewCompletedEvent` to Matching (via message broker)

## Tables

### interview_sessions

One session per candidate per application. Created when `CandidateReadyForInterviewEvent` is received. This is the top-level record that tracks the interview from start to finish.

### response_evaluations

The AI's assessment of an individual response. Created during the Processing phase after the candidate has submitted the entire interview. One-to-one with `interview_responses` using a shared primary key.

This table is the scoring layer — separate from the candidate's raw answer. If scoring fails or needs to be re-run (prompt tuning, provider change), the evaluation record can be deleted and recreated without touching the original response.

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| response_id | uuid | PK, FK → interview_responses.id | Shared key — one evaluation per response |
| score | decimal(5,2) | NOT NULL | AI-assigned score for this response (0.00–100.00) |
| score_reasoning | text | NOT NULL | AI's explanation of why this score was given. Transparency and debugging |
| topics_covered | jsonb | nullable | Which expected topics the candidate actually addressed. Compared against `expected_topics` on the question |
| evaluated_at | timestamp | NOT NULL | When the AI scored this response |
| created_at | timestamp | NOT NULL | |

No additional indexes — accessed via the PK join on `response_id`. The primary access pattern is loading all evaluations for a session by joining through responses.

### interview_evaluations

Comprehensive evaluation produced after all responses are scored. This is the final output — the data that gets sent back to the monolith's Matching module via `InterviewCompletedEvent`.

One-to-one with `interview_sessions` using a shared primary key.

|--------|------|-------------|-------------|
| session_id | uuid | PK, FK → interview_sessions.id | Shared key — one evaluation per session |

| technical_score | decimal(5,2) | NOT NULL | Technical knowledge and problem-solving ability (0.00–100.00) |
| communication_score | decimal(5,2) | NOT NULL | Clarity, articulation, and presentation (0.00–100.00) |
| experience_score | decimal(5,2) | NOT NULL | Depth and relevance of professional experience (0.00–100.00) |

| recommendation | varchar(20) | NOT NULL | Enum: `StrongAdvance`, `Advance`, `Borderline`, `DoNotAdvance`. AI's recommendation on whether to shortlist |
| strengths | jsonb | NOT NULL | Array of strings — key strengths identified across all responses |
| concerns | jsonb | NOT NULL | Array of strings — concerns or weaknesses identified |
| summary | text | NOT NULL | AI-generated narrative summary of the candidate's interview performance |
| score_breakdown | jsonb | NOT NULL | Per-category score breakdown with weights. See Score Breakdown Format below |

| created_at | timestamp | NOT NULL | |

## Relationships

```
interview_sessions ||--o{ interview_questions : "has many"
interview_sessions ||--o{ interview_responses : "has many (denormalized)"
interview_questions ||--o| interview_responses : "has (optional, one-to-one)"
interview_responses ||--o| response_evaluations : "has (optional, one-to-one)"
interview_sessions ||--o| interview_evaluations : "has (optional, one-to-one)"
interview_sessions ||--o{ ai_api_logs : "has many (one per API call)"
```

**No FKs to the monolith's database.** `tenant_id`, `application_id`, `job_posting_id`, and `applicant_id` are stored as plain UUIDs — not foreign keys. This service cannot and should not reference the monolith's tables. The IDs are correlation keys used to route events back to the correct tenant and application.

---

## Tenant Isolation

This is the one database in the system that uses **tenant ID filtering** instead of database-per-tenant. Every query must include `tenant_id` in the WHERE clause. The `ix_sessions_tenant_id` index and composite indexes with `tenant_id` as the leading column ensure this is efficient.

**Why not database-per-tenant here?** The AI Interview Service is stateless and compute-heavy. Interview data is short-lived (generated, scored, results sent back, eventually archived). Provisioning and managing a separate database per tenant for this service would add operational complexity for data that's mostly transient. A single database with tenant filtering is simpler and sufficient at the expected scale.

**Data in this database is a copy, not the source of truth.** The monolith's Matching module receives the `InterviewCompletedEvent` with the evaluation scores and stores what it needs in `candidate_matches`. The AI Interview DB is authoritative for interview details (questions, responses, scoring reasoning) but the pipeline decisions are made in the monolith.

## Interview Session Lifecycle

```
Pending → QuestionGeneration → InProgress → Processing → Completed
                                           → Expired
       → Failed (at any stage)
```

- **Pending**: Event received, session created. Waiting for question generation to begin.
- **QuestionGeneration**: AI is generating questions based on job requirements. Candidate can't start yet.
- **InProgress**: Questions generated, candidate is answering. `started_at` set. No scoring happens during this phase.
- **Processing**: Candidate has submitted all answers. `completed_at` set. Backend is transcribing (if voice/video), scoring responses, and generating the evaluation. Candidate is done — this is fully async.
- **Completed**: All processing finished. Evaluation generated. `InterviewCompletedEvent` published.
- **Expired**: Candidate didn't complete within `expires_at` deadline. Partial responses may be processed if any exist, depending on tenant config.
- **Failed**: Technical failure at any stage (AI provider error, timeout). Can be retried.

## Interview Flow

```
Phase 1 — Setup (async, before candidate starts):
1. CandidateReadyForInterviewEvent received from message broker
   - Payload: tenant_id, application_id, job_posting_id, applicant_id,
              job_title, required_skills, nice_to_have_skills
2. interview_sessions row created (status = Pending)
3. Question generation:
   a. Status → QuestionGeneration
   b. AI provider called with job requirements, title, skills
   c. Questions generated (mix of Technical, Behavioral, Situational, etc.)
   d. interview_questions rows created
   e. total_questions set on session
4. Session ready → Status → InProgress
   Candidate is notified that their interview is available

Phase 2 — Candidate answering (real-time, no blocking):
5. Candidate accesses interview via API (JWT-authenticated)
   Questions presented one at a time (or all at once — tenant-configurable)
6. Candidate answers each question:
   a. interview_responses row created (response_text for text, media_url for voice/video)
   b. questions_answered incremented on session
   c. NO scoring happens here — candidate moves to next question immediately
7. Candidate submits the interview (or answers the last question)
   a. Status → Completed, completed_at set

Phase 3 — Processing (async, candidate is done):
8. Transcription (voice/video responses only, parallelizable):
   a. For each response where response_type = Voice or Video:
      - transcription_status → Pending
      - Speech-to-text API called
      - Transcript stored in response_text
      - transcription_status → Completed, transcribed_at set
      - Logged in ai_api_logs (call_type = Transcription)
   b. On failure: transcription_status → Failed, can be retried

9. Scoring (all responses, parallelizable after transcription):
   a. For each response (in parallel):
      - AI scores the response_text against the question's expected_topics
      - response_evaluations row created (score, score_reasoning, topics_covered)
      - Logged in ai_api_logs (call_type = ResponseScoring)

10. Evaluation generation:
    a. AI generates comprehensive evaluation from all response scores
    b. Sub-scores computed per category (technical, communication, experience, behavioral)
    c. Overall score computed from weighted sub-scores
    d. Strengths, concerns, narrative summary generated
    e. interview_evaluations row created
    f. Logged in ai_api_logs (call_type = EvaluationGeneration)

11. InterviewCompletedEvent published to message broker:
    - Payload: tenant_id, application_id, overall_score,
               technical_score, communication_score, experience_score,
               recommendation, strengths, concerns, summary

12. Monolith's Matching module consumes event → updates candidate_matches
```

**Why batch scoring after completion, not per-response?**
- The candidate's experience is uninterrupted — no waiting for AI scoring between questions
- Transcription and scoring can be parallelized across all responses simultaneously
- A single evaluation call has the full context of all answers, which produces better overall assessment
- If one scoring call fails, it doesn't block the candidate mid-interview — it's retried in the background
- Reduces real-time latency requirements on the AI provider — async processing is more tolerant of slow responses

## AI API Audit & Cost Tracking

The `ai_api_logs` table provides a complete audit trail for every AI provider call. Each interview session generates multiple API calls:

```
Typical video interview (10 questions):
  1× QuestionGeneration     — generate all questions in one call
  10× Transcription         — speech-to-text for each video/voice response
  10× ResponseScoring       — score each transcribed response
  1× EvaluationGeneration   — produce the final evaluation
  ─────────────────────────
  22 total API calls logged (fewer for text-only interviews — no transcription step)
```

**What's tracked per call:**
- Token usage (input, output, total) for cost attribution
- Estimated cost in USD computed at call time using current model pricing
- Latency for performance monitoring and SLA tracking
- Success/failure status with error messages for debugging
- Retry count to detect flaky provider behavior
- Request/response summaries (metadata only — no PII or full prompts)

**What this enables:**
- **Cost attribution per tenant** — query `ai_api_logs` by `tenant_id` to see how much AI usage each tenant drives. Essential for usage-based billing or subscription tier enforcement
- **Provider comparison** — compare latency, error rates, and cost across providers/models if running A/B tests
- **Failure analysis** — find patterns in failed calls (time of day, specific models, specific question types)
- **Usage dashboards** — total tokens consumed, cost per interview, cost trends over time
- **Compliance** — prove that AI evaluations were generated by specific models at specific times, with traceable inputs and outputs

**What's NOT stored:**
- Full prompts (may contain candidate PII, job details, or proprietary scoring logic)
- Full AI responses (stored on the relevant entity — questions, response scores, evaluations)
- Raw HTTP request/response bodies

The `request_summary` and `response_summary` JSONB columns contain only metadata — enough to understand what happened without exposing sensitive data.

## Data Retention

Interview data in this database is detailed but becomes less valuable over time. Once the evaluation is sent to the monolith (via `InterviewCompletedEvent`) and the candidate progresses or is rejected, the detailed question/response/scoring data is primarily useful for:
- Auditing (why did the AI score this way?)
- Improving question generation and scoring prompts
- Dispute resolution (candidate challenges the AI evaluation)

A retention policy can archive or delete old sessions after a configurable period (e.g., 90 days after completion). The monolith retains the evaluation scores permanently in `candidate_matches`.

**Media files (audio/video) have storage cost implications.** Video recordings are significantly larger than text data. A separate, shorter retention policy for media files in blob storage is recommended (e.g., 30 days after evaluation completes). The transcripts remain in the database as `response_text` — the original recordings are only needed for playback and dispute resolution.

**API logs have a separate retention policy.** Cost and usage data in `ai_api_logs` may need to be retained longer than interview details (e.g., 12 months for billing reconciliation). The `ix_api_logs_called_at` index supports efficient time-range deletion for cleanup.

## Design Decisions

**Single shared database with tenant ID filtering.** Unlike the monolith's database-per-tenant approach, this service uses one database for all tenants. Interview data is transient and compute-heavy — the operational overhead of managing per-tenant databases for short-lived data isn't justified. Tenant isolation is enforced via `tenant_id` filtering on every query and composite indexes with `tenant_id` as the leading column.

**Denormalized job data on the session.** `job_title` and `job_requirements` are copied from the monolith's event payload, not looked up dynamically. The AI Interview Service can't query the monolith's database, and it needs this data for question generation. Denormalization is the correct pattern for cross-service data — the service owns a local copy of what it needs.

**Plain UUIDs for cross-service references, not FKs.** `tenant_id`, `application_id`, `job_posting_id`, and `applicant_id` are correlation keys, not foreign keys. This service has no access to the monolith's databases, so FK constraints are impossible. The IDs are used to route events back and to validate JWT claims.

**`response_type` enum covers all input methods.** `Text`, `Voice`, `Video`, `Skipped`, `TimedOut`. The type determines the processing pipeline: text goes straight to AI scoring, voice/video go through transcription first, skipped/timed-out get scored as zero or partial. Tracking this explicitly makes the flow clear and auditable.

**`response_text` is the universal scoring input.** Regardless of how the candidate responded, the AI scorer always works from text. For typed answers, it's what they wrote. For voice/video, it's the speech-to-text transcript. This means the scoring logic is identical across response types — no branching for "is this a video response?" in the scorer. The media files are for playback and audit, not for scoring.

**Transcription as a tracked processing step.** `transcription_status` (Pending/Completed/Failed) and `transcription_provider` make the speech-to-text pipeline visible and debuggable. If transcription fails, the response is stuck — it can't be scored until the transcript exists. This is analogous to how resume parsing works in the Profiles module: upload → async processing → data available for downstream use.

**Media files in blob storage, not the database.** Audio and video files are large. `media_url` points to CDN/blob storage. The database stores metadata only (URL, type, duration). Same pattern as resumes and documents throughout the system.

**AI provider and model tracked in API logs only.** Rather than duplicating `ai_provider` and `ai_model` on sessions and evaluations, the `ai_api_logs` table is the single source of truth for which provider and model was used for each operation. Need to know what generated the questions? Query logs by `session_id` and `call_type = 'QuestionGeneration'`. Need to know what produced the evaluation? `call_type = 'EvaluationGeneration'`. This avoids redundancy and handles edge cases like fallback providers cleanly — the logs show exactly what happened per call.

**Per-call API logging, not per-session summaries.** Individual call logs are more useful than aggregated session-level totals. You can always aggregate up (sum tokens per session), but you can't disaggregate down (which specific call failed? which scoring call was slowest?). The per-call granularity is worth the extra rows.

**Summaries instead of full prompts/responses in API logs.** Full prompts may contain candidate PII (resume text, personal details) and proprietary scoring criteria. Full responses are already stored on their target entities (questions, scores, evaluations). The API log captures metadata — enough for cost tracking and debugging without creating a PII liability.

**Four-level recommendation scale.** `StrongAdvance`, `Advance`, `Borderline`, `DoNotAdvance`. Aligned with the final interview's recommendation scale (`StrongHire`, `Hire`, `NoHire`, `StrongNoHire`) but using advancement language since this is a mid-pipeline recommendation, not a final hiring decision.

**`score_reasoning` on individual response evaluations.** Transparency into AI scoring is critical for trust. If a candidate or hiring manager questions why a response scored poorly, the reasoning text provides the AI's explanation. This is especially important for compliance and fairness auditing.

**Separate `response_evaluations` table, not scoring columns on responses.** The response record is the candidate's raw input — immutable once submitted. The evaluation is the AI's assessment — a separate concern that happens asynchronously during processing. Splitting them means: responses are never updated (clean append-only pattern), failed scoring can be retried by deleting and recreating the evaluation without touching the original response, and re-scoring with a different model or prompt is a clean insert, not an update that overwrites the previous assessment.

**Shared primary key for evaluations.** One evaluation per session, enforced at the database level. Same pattern used throughout the project.

**`session_id` denormalized on responses.** Responses already link to questions, which link to sessions. But the most common query is "all responses for this session" — denormalizing `session_id` avoids joining through `interview_questions` for this hot path.

**Expiration handling.** Interviews have a deadline (`expires_at`). If the candidate doesn't complete in time, the session expires. Partial responses (if any) can still be scored — a half-completed interview with strong answers is more signal than no interview at all. Whether to generate a partial evaluation is a tenant-configurable policy.
